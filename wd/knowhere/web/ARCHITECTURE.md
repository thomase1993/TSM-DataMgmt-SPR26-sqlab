# Web UI — Architecture & Data Flow

This document explains how the Knowhere web UI works end-to-end: where data comes from, how sessions are managed, and how the frontend and backend communicate.

---

## Overview

```
Browser (HTML + JS)
      │  HTTP (fetch)
      ▼
FastAPI (app.py, port 8080)
      │  python-duckdb
      ▼
In-memory DuckDB (per browser tab)
      │  initialized from
      ▼
output/dump.sql  ←  generated once by the sqlab CLI
```

The server is stateless in terms of storage — there are no files written at runtime. Every browser tab gets its own isolated in-memory DuckDB database.

## Where `dump.sql` Comes From

`output/dump.sql` is **pre-generated by the sqlab CLI** and committed to the repository. Students never need to regenerate it — it is present after `git clone`.

The file was created by running:
```bash
docker run --rm -v .:/workspace sqlab /workspace create
```

It contains, in order:
1. `CREATE TABLE` statements for `brokers` and `trades`
2. `CREATE PROPERTY GRAPH trade_graph` (requires duckpgq)
3. sqlab infrastructure tables (`sqlab_msg`, `sqlab_metadata`)
4. Scalar macros: `nn`, `string_hash`, `encrypt`, `decrypt`
5. `salt_NNN` macros (one per challenge token)
6. `INSERT` statements for all rows — **with `hash = 0`** (see below)
7. Encrypted challenge messages inserted into `sqlab_msg`
8. Game metadata in `sqlab_metadata`

### Why `hash = 0` in the dump

The sqlab CLI computes row hashes using `UPDATE … SET hash = string_hash(…)` after each INSERT, but does not write those `UPDATE` statements into `dump.sql`. As a result, all rows in the dump have `hash = 0`.

`app.py` recomputes the hashes at session-init time using `_update_hashes()`, which runs the same `string_hash(CONCAT('table:', name, '|', …))` formula the CLI uses. Without this step, every token formula (`salt_NNN(sum(nn(hash)) OVER ())`) evaluates to `0` and no challenge can be unlocked.

## Session Model

Each browser tab gets a **dedicated in-memory DuckDB connection**:

```python
sessions: dict[str, duckdb.DuckDBPyConnection] = {}
```

On first load (`GET /api/session`), the server:
1. Creates a new `duckdb.connect(":memory:", config={"allow_unsigned_extensions": True})`
2. Calls `_load_duckpgq()` — installs/loads the PGQ extension (see below)
3. Executes every statement from `dump.sql` via `split_statements()`
4. Runs `_update_hashes()` to populate the `hash` columns
5. Returns a UUID session ID stored in `localStorage`

On subsequent requests the session ID is sent with every API call and the existing connection is reused. If the server restarts the session is gone — the browser will automatically get a fresh one on the next API call.

**Isolation benefit:** DDL statements like `CREATE OR REPLACE PROPERTY GRAPH` in Challenge 2 affect only the current tab's session, so each student works independently.

---

## duckpgq Extension Loading

`CREATE PROPERTY GRAPH` and `GRAPH_TABLE` require the `duckpgq` extension. It is not bundled in the Docker image (the binary is platform-specific: `linux_amd64` vs `linux_arm64`). Instead, `_load_duckpgq()` tries three sources at runtime:

| #   | Source                                                                                             | Notes                                             |
| --- | -------------------------------------------------------------------------------------------------- | ------------------------------------------------- |
| 1   | `LOAD duckpgq`                                                                                     | Succeeds if already cached in the named volume    |
| 2   | `INSTALL duckpgq FROM community`                                                                   | DuckDB community repo (may 404 for some versions) |
| 3   | `SET custom_extension_repository = 'http://duckpgq.s3.eu-north-1.amazonaws.com'` + `FORCE INSTALL` | S3 mirror used by the sqlab project               |

The extension is saved to `/root/.duckdb/extensions/` inside the container, which is backed by the Docker named volume `knowhere_duckdb-extensions`. After the first successful download it is never re-downloaded.

## API Endpoints

| Method | Path                   | Purpose                                                                       |
| ------ | ---------------------- | ----------------------------------------------------------------------------- |
| `GET`  | `/`                    | Serves `static/index.html`                                                    |
| `GET`  | `/static/*`            | Static assets (volume-mounted, changes take effect on reload)                 |
| `GET`  | `/api/session`         | Creates a new in-memory DuckDB session, returns `{session_id}`                |
| `POST` | `/api/query`           | Executes SQL, returns `{columns, rows, session_id}` or `{error}`              |
| `GET`  | `/api/decrypt/{token}` | Calls `SELECT msg FROM decrypt(token)`, returns `{message, hint, session_id}` |

### `/api/query` — multi-statement handling

The submitted SQL is split into individual statements by `split_statements()` (respects single-quoted strings so base64 content in `sqlab_msg` is never split incorrectly). All statements except the last are executed silently; the last one's result set is returned to the browser.

### `/api/decrypt/{token}` — hint injection

The decrypted message comes entirely from the `decrypt` TABLE macro in `dump.sql`. The `hint` field is **not** in the database — it is looked up from `records.json` (mounted read-only at `/app/records.json`) keyed by the entry token of the challenge. This keeps hints out of the encrypted SQL dump.

## Frontend Logic (`app.js`)

```
DOMContentLoaded
  │
  ├── initSession()         GET /api/session if no session_id in localStorage
  └── decryptAndShow(42)    load the first challenge (token 42 is hardcoded as entry point)
          │
          └── populates left panel with mission text + hint button

▶ Run Query
  │
  └── POST /api/query  →  renderResults()
          │
          ├── error  →  red banner
          ├── no columns  →  "Statement executed successfully"
          └── table  →  render HTML table
                │
                └── token column detected?
                      └── yes  →  show "Unlock Next ▶" button

Unlock Next ▶
  │
  └── GET /api/decrypt/{token}
          │
          ├── starts with 🔴 (wrong answer)  →  amber banner in results, mission panel untouched
          └── valid mission  →  update left panel, clear editor, show hint button
```

**Wrong-answer detection:** the `decrypt` macro returns a fallback message starting with `🔴` when no encrypted message matches the token. The frontend intercepts this and shows a friendly nudge instead of overwriting the mission text.

## Docker Setup

```yaml
# docker-compose.yml (simplified)
services:
  web:
    build: ./web          # Dockerfile: python:3.12-slim + pip install
    ports: ["8080:8080"]
    volumes:
      - ./output/dump.sql:/app/dump.sql:ro        # DB snapshot
      - ./records.json:/app/records.json:ro        # hints & metadata
      - ./web/static:/app/static:ro               # live-reload for development
      - duckdb-extensions:/root/.duckdb/extensions # duckpgq cache

volumes:
  duckdb-extensions:  # persists across restarts; downloaded once per machine
```

The `static/` directory is volume-mounted so changes to `app.js`, `style.css`, or `index.html` are visible on the next browser refresh without rebuilding the image. Only changes to `app.py` or `requirements.txt` require `docker compose up --build`.
